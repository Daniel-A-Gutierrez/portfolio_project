## First order of business : Importing Pico

I made the initial mistake of not looking for scss specific instructions for this - 
the documentation says to simply do 
``` scss
    @use "pico"
```

Yeah its not that simple, but after lots of tinkering I've landed at the following
``` scss
	@use "@picocss/pico/scss/pico" with 
	(
		$theme-color: "violet", 
		$parent-selector:".pico",
		$breakpoints: (
			sm: (root-font-size: 85%),
			md: (root-font-size: 90%,),
			lg: (root-font-size: 95%),
			xl: (root-font-size: 100%,),
			xxl: (root-font-size: 105%,)
  		),);
```
Despite the specification of parent selector, which normally makes pico only affect things 
inside of a '.pico' classed container, typography seems to be the exception to that rule, so
overriding some variables there is necessary. 

I've also discovered that astro makes no guarantees about rule ordering for css so variables
need to be overridden with ```!important```, which is kind of annoying.

### GLOBAL PAIN
Ok this has become a day long ardour. 
I share my pain so that others may not know it. 
All I wanted was to have types in my editor but use a CDN link for the p5 library on the
page that uses it. 

``` typescript title="env.d.ts"
interface Window {
	myFunction(): boolean;
}
```

This should make myFunction available on the global Window object.
The goal is to make the p5 module available here instead.

When the p5 article loads it should dynamically load the p5 library, add it to window,
which will take a div and populate it with the sketch once its loaded. 

Unfortunately this doesnt work. 
``` typescript title="env.d.ts"
/// <reference types="astro/client" />
import type p5 from 'p5';
interface Window {
    p5 : p5;
    hi : string;
};
```

However, this DOES, but it has no idea what the p5 type actually is.
``` typescript title="env.d.ts"
/// <reference types="astro/client" />
//import type p5 from 'p5';
interface Window {
    p5 : p5;
    hi : string;
};
```

I've tried including p5 in tsconfig.jsons 'types' property, using <code>global.d.ts</code> instead,
using <code>"///<reference types='p5'/>"</code>, using <code>Interface Window</code> in env.d.ts,  
none of them work. 

It's frustrating the the first solution above actually does work within env.d.ts, but that doesn't
extend to the rest of the project somehow. 

I've posted a stack overflow question [here](https://stackoverflow.com/questions/79794549/how-can-i-import-an-external-javascript-library-from-a-cdn-for-a-website-and-s) 
in case this gets fixed in the future, the solution may be there. 

...

Ok actually, its the next day and I had an idea - astro must be doing some preprocessing to env.d.ts
or something, so using global.d.ts instead and following some advice [here](https://stackoverflow.com/questions/42233987/how-to-configure-custom-global-interfaces-d-ts-files-for-typescript) 
I managed to get types into my article component with 

``` typescript title="global.d.ts"
export {};
import type p5 from 'p5';
declare global 
{
    interface Window {
        p5 : p5;
    };
}
```

Now in the article I have to actually import it from a URL. 
This would've been simpler if p5 didnt export a class, a module, a namespace, and an interface all with the same Name but still
Dancing around name collisions we get : 
```typescript
//@ts-ignore
import P5 from "https://cdnjs.cloudflare.com/ajax/libs/p5.js/2.0.5/p5.esm.min.js";
window.p5 = P5;
import type p5 from "p5";
let p5 = window.p5;
```

<Image src='../../images/UnevenWidths.png' maxwidth='500px' alt='Uneven images on the blog page.'/>

Damnit. the types between the library and my code dont line up. 

This actually wasn't that hard to figure out though, it was my local types that were wrong because of all the finagling
I just did with the p5 identifier. It wants the class, so I just have to use it as a class and ignore typescript's screaming.

Before : 
```typescript 
	//before
    const sketchInstance : p5 = new p5.p5( sketch, mydiv ); //their function is incorrectly typed

	//after 
	//@ts-ignore
	const sketchInstance : p5 = new p5( sketch, mydiv ); 
```

At long last! I don't have to host p5 from my little server anymore! Hooray!

Man, I'm glad I have a template for doing this now but I really don't want to have to mess with this again.

### Client Side Routing Pain
I discovered a bug in my p5 article while working on the previous endeavor where the p5 example wouldn't load if the page wasn't
the initial page load - if you navigated there from the homepage for example, it wouldn't load the example.

Apparently astro and flamethrower both have special properties - astro-data-rerun and data-reload - 
that force a script to rerun after the initial load, but neither worked. 
Furthermore the astro one forces the script to be inline, meaning you cant use typescript. 

The flamethrower documentation explicitly says that body scripts rerun on page load but
that seems to no longer be the case.

After faffing about with the p5 page for another half a day eventually I got it working by appending the 
following blurb at the end. 

```typescript
    console.log("drawing p5 test");
    drawP5Test();
    window.addEventListener('flamethrower:router:end' , drawP5Test);
```

It turns out this is a problem for other client side routers too...  
https://github.com/vercel/next.js/discussions/17919  

It seems pages preloaded with \<link> tags no longer rerun body scripts?
I suppose if the intent is to preserve state between page loads it sorta makes sense...
