# Types Of Macros 
Rust macros are one of the least stable parts of the language, and there's a bunch of different types.  
I'll give some examples, then we'll get into the weeds of writing our own.

## [macro_rules!](https://doc.rust-lang.org/rust-by-example/macros.html)  
<CodeSnippet>
```rust 
    macro_rules! s 
    {
        ($str : literal) => {$str.to_string()}
    }
```
</CodeSnippet>
The syntax on these gets crazy - they're pretty strictly limited to token manipulation so they're less functional 
than the other types, but that same crazy syntax ironically makes them easier to write for certain tasks once you've 
gotten the hang of it. 

## [Proc Macros](https://doc.rust-lang.org/reference/procedural-macros.html)   
Proc macros need to be defined in a separate crate than the one using them, with "proc_macro=true" set in the cargo.toml under the lib tag. 
They use [syn](https://docs.rs/syn/latest/syn/) and [quote](https://docs.rs/quote/latest/quote/) to manipulate 
[Token Streams](https://doc.rust-lang.org/proc_macro/struct.TokenStream.html) which are both the input and output of 
the macro.  
There's three types.  
### header!(Function-Like Macros)
<CodeSnippet>
```rust
    #[proc_macro]
    pub fn tokenize(tokens: TokenStream) -> TokenStream 
    {
        let iter = tokens.clone().into_iter();
        for i in iter 
        {
            println!("{:?}", i);
        }
        return tokens;
    }
```
</CodeSnippet>

These guys are some of the most powerful - they can take a parsed token stream, execute arbitrary code, 
and output a token stream in response. This one is useful for learning how rust parses things.  
### <div><div>#[derive(Trait)]</div><div>Derive macros </div></div>

<CodeSnippet filename="lib.rs">
```rust
    use proc_macro::TokenStream;
    use quote::quote;

    #[proc_macro_derive(Greet)]
    pub fn greet_derive(input: TokenStream) -> TokenStream
    {
        let ast = syn::parse::<syn::DeriveInput>(input).unwrap();
        impl_greet_macro(&ast);
    }

    fn impl_greet_macro(ast :&syn::DeriveInput) -> TokenStream
    {
        let name = ast.ident;
        let generated = quote!{
            impl Greet for #name {
                fn greet(){ println!("Hello, Macro! My name is {}!", stringify(#name) ); }
            }
        };
        return generated.into();
    }
```
</CodeSnippet>

All rust programmers have used these extensively, but now we'll be learning how to write them outselves.
The process has you create a second external crate in addition to the first to hold your traits, but 
the input is easier to predict since they only annotate structs, enums, and unions. 

### <div><div>#[CustomAttribute(args)]</div><div>Attribute Macros</div></div>

<CodeSnippet>
```rust
    #[proc_macro_attribute]
    pub fn limit_recursion(attr: TokenStream, item: TokenStream) -> TokenStream {
        let limit = syn::parse_macro_input!(attr as syn::LitInt);
        let input = syn::parse_macro_input!(item as syn::ItemFn);
        let fn_sig = &input.sig;
        let block = &input.block;

        let expanded = quote::quote! {
            #fn_sig {
                println!("Please don't recur more than {} times.", #limit);
                #block
            }
        };
        TokenStream::from(expanded)
    }
```
</CodeSnippet>

Attribute macros can annotate basically anything, but they're often used on functions to modify their behavior 
in a way similar to a higher order function, but with the ability to inject code into the function itself. 

Axum's #[debug_handler] is the first example that comes to mind - it runs a bunch of assertions against your handler's 
code to generate compile time errors for common mistakes in using the library that might not show up as compiler errors otherwise.

They can also be troublesome due to name collisions with other variables in the scope.

### [CRAB TIME!](https://docs.rs/crabtime/latest/crabtime/) 
<CodeSnippet>
```rust 
    #[crabtime::function]
    fn gen_positions(name: String, components: Vec<String>) {
        for dim in 1 ..= components.len() {
            let cons = components[0..dim].join(",");

```
</CodeSnippet>
The previous example is from their docs.
Crab Time is a crate that lets you write function like procedural macros without the extra hassle of creating and maintaining 
a separate crate. It also lets you take in and output data the traditional way, or as normal rust types like Vec\<String>.

<CodeSnippet filename="macro use syntax tree">
```
    Syntax
    MacroRulesDefinition →
        macro_rules ! IDENTIFIER MacroRulesDef

    MacroRulesDef →
        ( MacroRules ) ;
        | [ MacroRules ] ;
        | { MacroRules }

    MacroRules →
        MacroRule ( ; MacroRule )* ;?

    MacroRule →
        MacroMatcher => MacroTranscriber

    MacroMatcher →
        ( MacroMatch* )
        | [ MacroMatch* ]
        | { MacroMatch* }

    MacroMatch →
        Tokenexcept $ and delimiters
        | MacroMatcher
        | $ ( IDENTIFIER_OR_KEYWORDexcept crate | RAW_IDENTIFIER | _ ) : MacroFragSpec
        | $ ( MacroMatch+ ) MacroRepSep? MacroRepOp

    MacroFragSpec →
        block | expr | expr_2021 | ident | item | lifetime | literal
        | meta | pat | pat_param | path | stmt | tt | ty | vis

    MacroRepSep → Tokenexcept delimiters and MacroRepOp

    MacroRepOp → * | + | ?

    MacroTranscriber → DelimTokenTree
```
</CodeSnippet>

MacroRulesDef is either "( MacoRules );" or "[ MacroRules ]" or "\{ MacroRules };"

MacroRules is a "MacroRule ( ; MacroRule) * ;?" which in the crazytalk of macro_rules! means "A macro rule, followed by at least 0 " ; MacroRule" patterns, and maybe a semicolon at the end."

## [Repetitions](https://doc.rust-lang.org/reference/macros-by-example.html#repetitions) 
Ok 2 lines in and we're learning new syntax.  
Similar to match statements, macro_rules destructure the input and allow you to use it in the body.
Repetitions are unique to macros though, and match - you guessed it - a repeated pattern in the input.

The general structure for input is $($name:type whatever else you want)[sep][*,+,?].
You can then use them in the output - without repeating ":type" - to repeat the entirety of the contents of the parenthesis for each 
instance of the pattern occurring in the input, separated by [sep]. 

The [*,+,?] tokens indicate how many times the pattern should repeat - any amount of times, at least once, or at most once. 

Here's an example. 

<CodeSnippet filename="autopsy">
```rust 
    // Define a macro to generate getter methods for a struct
    macro_rules! generate_getters {
        ($struct_name:ident, $($field:ident),*) => {
            impl $struct_name {
                $(
                    pub fn $field(&self) -> &str {
                        &self.$field
                    }
                )*
            }
        };
    }
```
</CodeSnippet>

The section inside the $() in the output, starting on line 5 and ending on line 9, will be repeated an amount of times equal to the 
destructured identifiers used inside it. In this case, equal to the number of times " $($field:ident),* " matched in the input. 

It's an error to use it with two paramters that don't match the same number of times. 

# Back To The Dissection

The inside of a macro rules is similar to a big match statement - or a case statement if you're 
not a rust native. Honestly props to you if you aren't, you have way too much faith in me
to make this digestible. Geniunely.

A MacroRule is "MacroMatcher=>MacroTranscriber", similar to how a match statement is made up of (Pattern)=>\{Block\}s ,

A MacroMatcher is     
<blockquote>
MacroMatcher →
          ( MacroMatch* )
        | [ MacroMatch* ]
        | \{ MacroMatch* }
</blockquote>
The syntax guide uses similar syntax to what its trying to teach us. 
It's any enclosing tags around any number of individual MacroMatch'es. 

Ok, a MacroMatch can be a list of things, including 
- any Token except Delimters ( () [] \{} ) and $ - meaning we could match on say, the number 3 distinctly from the number 4.
- A MacroMatcher which itself is a list of MacroMatches - so repeaters and/or recursive macros? .
- "$(IDENTIFIER_OR_KEYWORD except crate OR a raw identifier. OR _ )" : MacroFragSpec . I think we don't literally need the parens here. 
The broad categories of tokens are 
- Keywords 
- Identifiers 
- Literals 
- Lifetimes 
- Punctuation 
- Delimeters

And MacroFragSpec is a pattern of token/tokens, and is one of the following:  
block, expr, ident, item, lifetime, literal, meta, pat, pat_param, path, stmt, tt, ty, vis   

These 'types' are NOT mutually exclusive. 
- Expr is made of other expr which can include basically anything (hard to break up also), 
- ident's are keywords and variable/function names, 
- literals are "3.0" and other fixed values in the code, 
- tt is literally any token, or a group of tokens in delimters, 
- stmt is a line of code excluding the semicolon, 
- ty is a type 
- vis is 'pub' or the absence thereof. It doesn't need a ? when matching on it.  

I found this example in the reference for each type of MacroFragSpec 

<CodeSnippet filename="syntax">
```rust
    // Used as an expression.
    let x = vec![1,2,3];

    // Used as a statement.
    println!("Hello!");

    // Used in a pattern.
    macro_rules! pat {
        ($i:ident) => (Some($i))
    }

    if let pat!(x) = Some(1) {
        assert_eq!(x, 1);
    }

    // Used in a type.
    macro_rules! Tuple {
        { $A:ty, $B:ty } => { ($A, $B) };
    }

    type N2 = Tuple!(i32, i32);

    // Used as an item.
    thread_local!(static FOO: RefCell<u32> = RefCell::new(1));

    // Used as an associated item.
    macro_rules! const_maker {
        ($t:ty, $v:tt) => { const CONST: $t = $v; };
    }
    trait T {
        const_maker!{i32, 7}
    }

    // Macro calls within macros.
    macro_rules! example {
        () => { println!("Macro call in a macro!") };
    }
    // Outer macro `example` is expanded, then inner macro `println` is expanded.
    example!();
```
</CodeSnippet>

- MacroRepSep : is any token except delimiters, *, + , OR ? . Just a character / string for going between successive items I guess. 
- MacroRepOp :*, + , or ? - the weird characters. 
- MacroTranscriber : "DelimTokenTree" Ok really if its just going to be 1:1 is it really necessary to split them up...
- DelimTokenTree : It's basically "[anything]" inside of delimeters.

# Taking Apart That Macro 
<CodeSnippet filename="autopsy">
```rust 
    // Define a macro to generate getter methods for a struct
    macro_rules! generate_getters {
        ($struct_name:ident, $($field:ident),*) => {
```

Lets go through and label each part of this with what it is. 

- macro_rules! : hardcoded part of the macro rules definition. notably, not a macro
- generate_getters : IDENTIFIER 
- \{everything else}; : MacroRulesDef, the "everything else" is MacroRules 
- ($struct_name:ident, $($field:ident), *) : A macro matcher, the bits inside are MacroMatches.
- $struct_name:ident, : $IDENTIFIER:MacroFragSpec , we're matching on an identifier and calling it struct_name. 
- "$($field:ident), *" : The third arm of MacroMatch, a recursive list match. 
    We're expecting a comma separated list of at least 0 identifiers, and naming it "field"  
- => \{ MacroTranscriber } 
- This bit is going to be output more or less literally with the  
    $ identifying identifiers to use from the macromatchers 
- impl $struct_name \{ $(stuff)* } : will create a impl block for $struct_name with  
    a function for each $field 
- pub fn $field(&self) -> &str \{&self.$field} : the getter that takes "&self" and returns "&self.$field".

Notably this macro doesnt return the right type for the getter - it only ever returns &str, regardless of the type of $field.

Maybe we can try improving/fixing it?

# Syn
The syn crate defines basically the entirety of the syntax of rust as a gigantic type system. For example 

<CodeSnippet>
```rust
    //syn internals
    pub struct DeriveInput {
        pub attrs: Vec<Attribute>,
        pub vis: Visibility,
        pub ident: Ident,
        pub generics: Generics,
        pub data: Data,
    }
```
</CodeSnippet>

While DeriveInput isn't a normal part of the rust syntax, this pattern extends to everything else - struct definitions, function contents, match statements, everything.
Attrs is the outer and inner attributes that could appear on anything that a derive can be added to, vis is the 'pub' or abscence thereof, 'ident' is the name of the struct, 
generics include the names and bounds, data is the fields which are the names and types and generics, etc. 

All of these types implement parse, which takes a ParseStream - convertible from a TokenStream with .into() - and either advance its internal pointer and yield themselves, or 
return an err if the next thing in the parsestream is not themself. 

For example you could do something like this to attempt to parse the 'struct' keyword out from a TokenStream. 
Don't actually do that though, theres a better way. 

<CodeSnippet>
```rust
    fn macro(tokens:TokenStream)
    {
        //token! makes a struct that implements parse
        let strct = Token!(struct).parse(tokens.into());
    }
```
</CodeSnippet>

# The Parse Trait
The correct way, as far as I'm concerned, to write a proc macro is to define your input as a structured type like syn does for the entirety of rust, then implement 
the parse trait from syn for your input types, then simply parse your input using the provided macro like so. 

<CodeSnippet>
```rust

    use proc_macro::TokenStream;
    use syn::{parse_macro_input, Result};
    use syn::parse::{Parse, ParseStream};

    struct MyMacroInput {
        /* ... */
    }

    impl Parse for MyMacroInput {
        fn parse(input: ParseStream) -> Result<Self> {
            /* ... */
        }
    }

    #[proc_macro]
    pub fn my_macro(tokens: TokenStream) -> TokenStream {
        let input = parse_macro_input!(tokens as MyMacroInput);

        /* ... */
    }
```
</CodeSnippet>

# Quote! 

Ok there's a bit to this but its not too complicated. Besides quote!, the 
documentation mentions a few other things : 

- quote_spanned! : You can get the span of basically anything from syn or proc_macro,   
    ie where it occurs in code, and use quote_spanned to make errors appear there when they occur in the macro.
- prettyplease can be used to make generated code look nice.
- format_ident!() : like format but it can take in ident's and outputs the modified name!  
This is what I needed for making my getters earlier. 

Ok onto the meat : 

Repetitions! 

In macro_rules we used $($var anything else)sep[*,+,?] in the matcher then the same
syntax in the output. 

With quote we do substitutions, aka interpolation, with #var instead, and the repetition 
syntax is similarly #(#var anything else)sep[*,+,?] . 

The difference is that now we're pulling from iterators we can take into the macro as arguments 
or create there as data. 

For example we could do  ...

<CodeSnippet filename="proc_macros/lib.rs">
```rust 
    fn impl_greet_macro(ast :&syn::DeriveInput) -> TokenStream
    {
        let name = &ast.ident;
        let rname = std::iter::repeat_n(&ast.ident,4);
        let generated = quote!{
            impl Greet for #name {
                fn greet(){
                    #(println!("Hello, Macro! My name is {}!", stringify!(#rname) );)*
                }
            }
        };
        return generated.into();
    }
```
</CodeSnippet>
Any iterable can be repeated on using 
the # version of the macro_rules! like syntax from quote!

# Other Need To Know's About quote!

We're allowed to use multiple helper functions and combine their outputs together using quote! like so 

<CodeSnippet>
```rust 
    let type_definition = quote! {...};
    let methods = quote! {...};

    let tokens = quote! {
        #type_definition
        #methods
    };
```
</CodeSnippet>

Calling functions and trait methods should be done in a particular way : 

<CodeSnippet>
```rust 
    quote! {let value = #field_type::new()} // bad 
    quote! {let value = <#field_type>::new();} //good 
    quote! {let value = <#field_type as core::default::Default>::default();} //for trait methods like Default
```
</CodeSnippet>

Comments dont get interpolated variables, but you can do 
<CodeSnippet>
```rust 
    #[doc = "try to interpolate: #ident"]
```
</CodeSnippet>

And lastly, if you're trying to interpolate tuple indeces, use 
<CodeSnippet>
```rust 
    let i = 0usize..self.fields.len(); //bad, results in "tuple.0usize.heap_size()
    let i = (0..self.fields.len()).map(syn::Index::from); 

    // expands to 0 + self.0.heap_size() + self.1.heap_size() + ...
    quote! {
        0 #( + self.#i.heap_size() )*
    }
```

# Parse
So it seems like each type in syn - almost anyway - implements the parse trait, 
which has the parse method. ParseStream also has a parse method but not from the trait.

Instead, the one on ParseStream is 

<CodeSnippet>
```rust
    pub fn parse<T: Parse>(&self) -> Result<T> {
        T::parse(self)
    }
```
</CodeSnippet>

So it just uses the one from whatever you're trying to parse from it. 
It also says that it advances the internal buffer even though it takes 
a & instead of &mut, so it should be fine to use it successively to take 
the things we want from the parse stream. 

The caveat seems to be that leaving the input token stream not-fully-read 
constitutes an error in the macro, so the way the syn crate demonstrated 
earlier is probably best.

<CodeSnippet>
```rust
    struct EvalInput
    {
        comp : Vec<OpOrLit>,
        comma : Token!(,),
        ty : syn::Type
    }

    enum OpOrLit
    {
        BOp(syn::BinOp),
        UOp(syn::UnOp),
        LitF(syn::LitFloat),
        LitI(syn::LitInt),
        ID(syn::Ident)
    }


    impl Parse for OpOrLit
    {
        fn parse(input: ParseStream) -> Result<Self>
        {
            if Ok(res)=input.parse<syn::BinOp>()        {Ok(OpOrLit::BOp(res))}
            else if Ok(res)=input.parse<syn::UnOp>()    {Ok(OpOrLit::UOp(res))}
            else if Ok(res)=input.parse<syn::LitFloat>(){Ok(OpOrLit::LitF(res))}
            else if Ok(res)=input.parse<syn::LitInt>()  {Ok(OpOrLit::LitI(res))}
            else if Ok(res)=input.parse<syn::Ident>()   {Ok(OpOrLit::ID(res))}
            else {Err("Fuck")}
        }
    }

    impl Parse for EvalInput
    {
        fn parse(input:ParseStream) -> Result<Self>
        {
            let terms = vec![];
            while let Ok(op_or_lit) = OpOrLit::parse(input) {terms.push(op_or_lit);}

            return EvalInput{ comp: terms, 
                            comma: input.parse<Token!(,)>()?, 
                            ty: input.parse<syn::Type>()?};
        }
    }

    #[proc_macro]
    pub fn eval_as(tokens: TokenStream) -> TokenStream 
    {
        let mut input = parse_macro_input!(tokens as EvalInput);
        let mut output = quote!{};
        let ty = input.ty;
        for term in input.comp
        {
            let o = match term{
                OpOrLit::LitF(x) => quote!{#x as #ty},
                OpOrLit::LitI(x) => quote!{#x as #ty},
                OpOrLit::ID(x) => quote!{#x as #ty},
                x => quote!{#x}
            };
            output = quote!{#output #o};
        }
        return output.into();
    }
```

# Other Tidbits From The Little Book Of Macros 
- proc_macros2 isnt a successor, its just a proc_macro crate that can be used outside of crates with proc_macro set in cargo.toml.
