# When Are Macros Useful 

- When you need to take an arbitrary number or type of arguments. 
- When you want to avoid writing / updating large amounts of repetitive code.
- Automaticallly implementing traits with derive macros
- Doing wacky shit that's not even really rust anymore.
- Doing basically anything you want at compile time, like writing files or api requests. 
- Injecting behavior (such as performance logging) for functions.

I'll make each of those a goal, and consider this done when I've done all of them.

then start a project and try running those examples, modifying them, then finally

<CodeSnippet filename="macro use syntax tree">
```

    MacroRulesDefinition →
        macro_rules ! IDENTIFIER MacroRulesDef

    MacroRulesDef →
        ( MacroRules ) ;
        | [ MacroRules ] ;
        | { MacroRules }

    MacroRules →
        MacroRule ( ; MacroRule )* ;?

    MacroRule →
        MacroMatcher => MacroTranscriber

    MacroMatcher →
        ( MacroMatch* )
        | [ MacroMatch* ]
        | { MacroMatch* }

    MacroMatch →

```

# Items 
Ok i've stumbled upon something useful so I'm just going to leave it here.  
An item is something in a crate, and follows the pattern  
OuterAttribute* (VisItem | MacroItem) 

A VisItem is any item that can be 'pub' or not, including 
- Module
- ExternCrate
- UseDeclaration
- Function
- TypeAlias
- Struct
- Enumeration 
- Union
- ConstantItem
- StaticItem
- Trait
- Implementation
- ExternBlock 
A MacroItem is either a macro invocation, or a MacroRulesDefinition.

An OuterAttribute is something like a #[derive(Trait)] or [rustfmt::skip], it applies an attribute to the
thing following it. 

An InnerAttribute applies to the thing its in, like #![feature(SomethingUnstable)] in a crate's main.rs. 

# Taking Apart That Macro 
<CodeSnippet filename="autopsy">
```rust 
    // Define a macro to generate getter methods for a struct
    macro_rules! generate_getters {
        ($struct_name:ident, $($field:ident),*) => {
            impl $struct_name {
                $(
                    pub fn $field(&self) -> &str {
                        &self.$field
                    }
                )*
            }
        };
    }
```
</CodeSnippet>

Lets go through and label each part of this with what it is. 

- macro_rules! : hardcoded part of the macro rules definition. notably, not a macro
- generate_getters : IDENTIFIER 
- \{everything else}; : MacroRulesDef, the "everything else" is MacroRules 
- ($struct_name:ident, $($field:ident), *) : A macro matcher, the bits inside are MacroMatches.
- $struct_name:ident, : $IDENTIFIER:MacroFragSpec , we're matching on an identifier and calling it struct_name. 
- "$($field:ident), *" : The third arm of MacroMatch, a recursive list match. 
    We're expecting a comma separated list of at least 0 identifiers, and naming it "field"  
- => \{ MacroTranscriber } 
- This bit is going to be output more or less literally with the  
    $ identifying identifiers to use from the macromatchers 
- impl $struct_name \{ $(stuff)* } : will create a impl block for $struct_name with  
    a function for each $field 
- pub fn $field(&self) -> &str \{&self.$field} : the getter that takes "&self" and returns "&self.$field".

Notably this macro doesnt return the right type for the getter - it only ever returns &str, regardless of the type of $field.

Maybe we can try improving/fixing it?

    fn impl_greet_macro(ast :&syn::DeriveInput) -> TokenStream

        let name = ast.ident;
        let generated = quote!{
            impl Greet for #name {
                fn greet(){ println!("Hello, Macro! My name is {}!", stringify(#name) ); }
            }

Ok lets go over this. 
quote is the library that fullfills the funciton of the MacroTranscriber block in macro_rules! - 
it takes what we wrote and spits it out as a token stream.

This is a derive macro, so we define a trait and a macro that will automatically 
impl it for an Enum, Struct, or Union - thats greet_derive. 

It gets a token stream, which we use syn to attempt to parse into a DeriveInput.  
If that fails, we panic because we can't return results here. 

We then pass that to our own function that actually makes use of the DeriveInput and quote! to 
generate the macro output. 

Honestly so much simpler than macro_rules!.

# Other Need To Know's About quote!

We're allowed to use multiple helper functions and 
combine their outputs together using quote! like so 

<CodeSnippet>
```rust 
    let type_definition = quote! {...};
    let methods = quote! {...};

    let tokens = quote! {
        #type_definition
        #methods
    };
```
</CodeSnippet>

Calling functions and trait methods should be done in a particular way : 

<CodeSnippet>
```rust 
    quote! {let value = #field_type::new()} // bad 
    quote! {let value = <#field_type>::new();} //good 
    quote! {let value = <#field_type as core::default::Default>::default();} //for trait methods like Default
```
</CodeSnippet>

Comments dont get interpolated variables, but you can do 
<CodeSnippet>
```rust 
    #[doc = "try to interpolate: #ident"]
```
</CodeSnippet>

And lastly, if you're trying to interpolate tuple indeces, use 
<CodeSnippet>
```rust 
    let i = 0usize..self.fields.len(); //bad, results in "tuple.0usize.heap_size()
    let i = (0..self.fields.len()).map(syn::Index::from); 

    // expands to 0 + self.0.heap_size() + self.1.heap_size() + ...
    quote! {
        0 #( + self.#i.heap_size() )*
    }
```
</CodeSnippet>

# Parse

So it seems like each type in syn - almost anyway - implements the parse trait, 
which has the parse method. ParseStream also has a parse method but not from the trait.

Instead, the one on ParseStream is 

<CodeSnippet>
```rust
    pub fn parse<T: Parse>(&self) -> Result<T> {
        T::parse(self)
    }
```
</CodeSnippet>

So it just uses the one from whatever you're trying to parse from it. 
It also says that it advances the internal buffer even though it takes 
a & instead of &mut, so it should be fine to use it successively to take 
the things we want from the parse stream. 

The caveat seems to be that leaving the input token stream not-fully-read 
constitutes an error in the macro, so the way the syn crate demonstrated 
earlier is probably best.

<CodeSnippet>
```rust
    struct EvalInput
    {
        comp : Vec<OpOrLit>,
        comma : Token!(,),
        ty : syn::Type
    }

    enum OpOrLit
    {
        BOp(syn::BinOp),
        UOp(syn::UnOp),
        LitF(syn::LitFloat),
        LitI(syn::LitInt),
        ID(syn::Ident)
    }


    impl Parse for OpOrLit
    {
        fn parse(input: ParseStream) -> Result<Self>
        {
            if Ok(res)=input.parse<syn::BinOp>()        {Ok(OpOrLit::BOp(res))}
            else if Ok(res)=input.parse<syn::UnOp>()    {Ok(OpOrLit::UOp(res))}
            else if Ok(res)=input.parse<syn::LitFloat>(){Ok(OpOrLit::LitF(res))}
            else if Ok(res)=input.parse<syn::LitInt>()  {Ok(OpOrLit::LitI(res))}
            else if Ok(res)=input.parse<syn::Ident>()   {Ok(OpOrLit::ID(res))}
            else {Err("Fuck")}
        }
    }

    impl Parse for EvalInput
    {
        fn parse(input:ParseStream) -> Result<Self>
        {
            let terms = vec![];
            while let Ok(op_or_lit) = OpOrLit::parse(input) {terms.push(op_or_lit);}

            return EvalInput{ comp: terms, 
                            comma: input.parse<Token!(,)>()?, 
                            ty: input.parse<syn::Type>()?};
        }
    }

    #[proc_macro]
    pub fn eval_as(tokens: TokenStream) -> TokenStream 
    {
        let mut input = parse_macro_input!(tokens as EvalInput);
        let mut output = quote!{};
        let ty = input.ty;
        for term in input.comp
        {
            let o = match term{
                OpOrLit::LitF(x) => quote!{#x as #ty},
                OpOrLit::LitI(x) => quote!{#x as #ty},
                OpOrLit::ID(x) => quote!{#x as #ty},
                x => quote!{#x}
            };
            output = quote!{#output #o};
        }
        return output.into();
    }
```

# Other Tidbits From The Little Book Of Macros 
- proc_macros2 isnt a successor, its just a proc_macro crate that can be used outside of crates with proc_macro set in cargo.toml.
- syn can parse non-rust syntax with the generic parsing api 
- macro use isnt necessary for external crates but is for other modules in the same crate
One last thing before we call it a day.
- proc macros are unhygenic - they just dump their contents where theyre invoked, so make sure not to name anything they generate something indistinct.
