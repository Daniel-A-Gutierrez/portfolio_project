# Meta
Introductory level tutorials and articles often glaze over errors and the struggle involved
in producing the code therein, but learning to interpret the errors the compiler gives you
is the hardest and most important skill in learning a language, so to that end, I went out 
of my way to show as many common mistakes as possible, as well as including the multitide
of genuine mistakes the compiler helped me catch as I wrote the code for this tutorial. 

# What Makes SQLite Special?

SQLite is the most widely used database in the world - searching for it in the applications you
use on a daily basis is a bit like turning over a rock in the woods looking for bugs. It sits
at the core of the IOS filesystem, Web Browsers, and is available as an official plugin for 
the Unreal Engine. 

SQLite is unique from most other databases in that it is in-process, acting more like a
library rather than a distinct application or binary running on the client machine. 
SQLite databases are also constrained to a single file and thread, making them simple to manage 
and backup, while retaining many of the features and familiarity of SQL. 

The reason I love it though, is that its the quickest and easiest way to prototype a database
application. Here's how you can get started in rust.

# Initializing the Database

To start with, we just make a connection to a new in-memory database. 
Our table is going to have 4 columns - id, unixtime, key, value.
<CodeSnippet filename="main.rs">
```rust 
    async fn init_db() -> Result<Connection>
    {
        let conn = Connection::open_in_memory()?;
        //define schema
        return Ok(conn);
    }
```
</CodeSnippet>

This defines an async function which returns a future (implicitly because its async) that yields a Result\<Connection> - 
either Ok\<Connection> or Err\<Anyhow::Error>.

In rust async functions must be awaited on to run, unlike javascript which automatically
runs the future and only pauses execution of the current task when it is awaiting another. 
Running a task asynchronously from the current one requires a call to tokio::spawn(). 

Connection::open_in_memory also returns a result. 
The question mark operator signifies that should that be an error, return it. 

<CodeSnippet filename="main.rs">
```rust 
    async fn init_db() -> Result<Connection>
    {
        let conn = Connection::open_in_memory()?;
        //define schema
        conn.execute("CREATE TABLE IF NOT EXISTS Storage (id : INTEGER PRIMARY KEY AUTOINCREMENT, 
                    unixtime: INTEGER, key : UNIQUE TEXT, value : TEXT);
                    CREATE UNIQUE INDEX IF NOT EXISTS idx_storage_keys ON storage (key); ", ())?;    
        return Ok(conn);
    }
```
</CodeSnippet>

Here we make the table, and create an index on our key column so it can function efficiently as a KV store.
Our time is an integer because we're going to use the builtin unixepoch() function, which returns the number
of seconds since Jan 1 1970.

# Inserting Into Our KV Store

Lets first make a function that inserts a key value pair into our database. 
It should check and return different errors in the condition that 
1. the key or value are too big,
2. the key is already present,
3. something else went wrong.

It doesn't actually need to be async because rusqlite's in-memory database is fully synchronous. 
Lastly, it needs to take its arguments by reference to avoid unnecessarily copying or consuming them.

<CodeSnippet filename="main.rs">
```rust 
    enum SetError { Oversized, KeyAlreadyPresent, DBError(rusqlite::Error) }

    async fn db_set(conn : &Connection, key : &str, val : &str) -> Result<() , SetError>
    {
        if key.len() > 10_000 || val.len() > 10_000 { return Err(SetError::Oversized) };
        todo!();
    }
```
</CodeSnippet>

A decent start! Also note the todo!() macro - this will cause a panic if it runs but silences errors 
relating to the incompleteness of the function otherwise.

Lets continue with ```conn.execute("INSERT INTO storage (unixepoch(),$1,$2);", (key, val))?```. 
Key takes the place of $1, and val takes $2. However, we have an error :  

<h3><code>`?` couldn't convert the error to `SetError`</code></h3>

Theres 2 ways to solve this - we either implement the From trait for SetError to allow it to automatically
be converted by ? from a rusqlite error, or we map the error type. We'll do the latter since its only used 
this one time. 
<CodeSnippet filename="main.rs">
```rust 
    fn db_set(conn : &Connection, key : &str, val : &str) -> Result<() , SetError>
    {
        if key.len() > 10_000 || val.len() > 10_000 { return Err(SetError::Oversized) };

        conn.execute("INSERT INTO storage (unixepoch(),$1,$2);", (key, val))
        .map_err(
            |e| {
                match e.sqlite_error_code() {
                    Some(rusqlite::ErrorCode::ConstraintViolation) => SetError::KeyAlreadyPresent, 
                    _ => SetError::DBError(e),
                }
            })?;
        return Ok(());
    }
```
</CodeSnippet>
Map Error takes a closure - an inline function with syntax that looks like |args| \{implicit return value} - 
that expects to be given the error and returns a different kind of error. 

In our case, we are doing a match on the value of e.sqlite_error_code(). I don't know why this yields 
an option but regardless that means we have to wrap the error type we're looking for in Some(T) since an 
Option's variants are Some(T) and None. The T we're looking for is rusqlite::ErrorCode::ConstraintViolation,
and with the fat arrow we map it to our SetError enum. 

Anything other than that (signified with '_') we just wrap in SetError:DBError(e). 

Constraint Violation is technically any constraint but our table's only constraint is the uniqueness one, 
so its fair to assume that it means the key is already present. 

And thats about it for inserting into our KV Store! We cant test it till we make a way to retrieve values 
so lets do that next. 

# Reading The KV Store

First step - read some documentation. I found this function in the rusqlite crate that seems to do what we want, 
lets break it down. 
<CodeSnippet filename="main.rs">
    ``` rust 
    pub fn query_one<T, P, F>(&self, sql: &str, params: P, f: F) -> Result<T>
    where
        P: Params,
        F: FnOnce(&Row<'_>) -> Result<T>,
    ```
</CodeSnippet>
<i>
    <blockquote>
    "Convenience method to execute a query that is expected to return exactly one row.

    Returns Err(QueryReturnedMoreThanOneRow) if the query returns more than one row.

    Returns Err(QueryReturnedNoRows) if no results are returned. If the query truly is optional, you can call .optional() on the result of this to get a Result\<Option\<T>> (requires that the trait rusqlite::OptionalExtension is imported)."
    </blockquote>
</i>

It takes 3 generic type parameters.
- P is the simplest, its just a tuple of things that can be parameters to a prepared statement, just like the execute function we used earlier.
- T is the desired output of this function. We should make a KVRow struct to map our database storage table into rust. 
- F is a function or closure that takes in a &Row, and attempts to convert it to T, returning a result.

<CodeSnippet filename="main.rs">

In rust, structs are data, and traits are groups of functions associated with a struct.
You implement a trait for a struct by defining the trait's members.
The TryFrom\<T> trait has a type parameter T (what we're converting from) and 2 members :  
Error and try_from(value: T) -> etc.  . 

We can use try_from to go from a sqlite database row to some defined type in a clean idiomatic way. 
Here's my first crack at it :

We can safely take the compilers advice here and just wrap KVRow::try_from(row) in Ok(T) but as for why that works ...
In our try_from impl we specified that try_from returns an anyhow::Result, but F's definition says it returns a rusqlite::Result.
If we wrap it in Ok, we'll get a rustqlite::Result\<anyhow::Result\<KVRow>>, which is technically fine.

Since theres nested results though we'd need two '?' operators.  
This is kinda sloppy though, and our desired return type is a Result\<Option\<KVRow>>.
To get there we can use the OptionalExtension module of rusqlite. 

<CodeSnippet filename="row.optional()">
    ```

    Converts a Result<T> into a Result<Option<T>>.
    ```

Not quite there, we want that inner result to be collapsed. 
Backtracking a bit, we're going to use expect to unwrap the inner result from try_from. 
This will cause the program to panic! or basically crash at runtime if it fails to parse a row from storage into a KVRow,
while providing an error message of our specification.
In general, Results are best when you want to make use of conditional behavior at runtime based on whether something succeeds,
whereas panics are appropriate if the error shouldn't be recovered from or ignorable.

# Sharing State

We don't have a global dynamic state like javascript or python, so to share our database connection between routes we need to use axum::extract::State. 

<CodeSnippet filename="main.rs">
    ``` rust
        let db = init_db().expect("Couldn't connect to database");
        //let state = State(Arc::new(Mutex::new(db)));
        let api_router = axum::Router::new().route("/hello", get(hello))
                                            .with_state(db)
                                            .into_make_service();
    ```
</CodeSnippet>

Which yields the error :

<CodeSnippet filename="main.rs">
    ``` rust 
        the trait bound `rusqlite::Connection: Clone` is not satisfied
        the trait `Clone` is not implemented for `rusqlite::Connection`

        `RefCell<rusqlite::inner_connection::InnerConnection>` cannot be shared between threads safely
    ```
</CodeSnippet>

Some database connections will be cloneable, some not, depending on if they do 
synchronization internally. Rusqlite does not, and the RefCell means that we can't send
references to it between threads either without a proper synchronization mechanism like 
Mutex or RWLock as well as an Atomic Reference-Counted Cell to ensure it lives long enough. 

<CodeSnippet filename="main.rs">
    ``` rust 
        use std::{net::SocketAddr, sync::Arc};
        use tokio::{self, sync::Mutex};
        ...
        //main()
        let state = State(Arc::new(Mutex::new(db)));
        let api_router = axum::Router::new().route("/hello", get(hello))
                                            .with_state(state)
    ```
</CodeSnippet>

IMPORTANT NOTE : when using async, be sure to use tokio::sync 's version of mutex, rwlock, channel, etc. over 
std::sync, to prevent blocking worker threads or potentially causing deadlocks. 

For now the compiler is happy but there's still problems with this we'll have to deal with 
in a moment.

Lets change focus to getting data from the client into our handler function.  
Axum uses extractors for this. I want to use JSON for the api so lets take a look at the JSON extractor documentation.

<blockquote><i>
" ⚠️ Since parsing JSON requires consuming the request body, the Json extractor must be last if there are multiple extractors in a handler. See “the order of extractors ”
</i></blockquote>

Read more [here](https://docs.rs/axum/latest/axum/struct.Json.html). Mimicking the code in their examples we need to 
add a dependency on serde 

<CodeSnippet filename="bash">
    ``` bash
        cargo add serde --features derive
    ```
</CodeSnippet>

"use" it, and create a KVRequest struct that derives serde::Deserialize. 

<CodeSnippet filename="main.rs">
    ``` rust 
        use serde::Deserialize;
        ...
        #[derive(Deserialize)]
        struct KVRequest {key : String, value: String}
    ```
</CodeSnippet>

Now we can write our handler function header like so 

<CodeSnippet filename="main.rs">
    ``` rust 
        type AppState = State<Arc<Mutex<rusqlite::Connection>>>;
        ...
        async fn kv_set(state : AppState, req : Json<KVRequest>) -> impl IntoResponse
        {return ();}
    ```
</CodeSnippet>

Except not! This is actually a strange error...

<CodeSnippet filename="error">
    ```
        required by a bound in `__axum_macros_check_kv_set_0_from_request_check`rustcE0277
        example.rs(33, 25): original diagnostic
        the trait bound `Arc<tokio::sync::Mutex<rusqlite::Connection>>: FromRef<()>` is not satisfied
    ```
</CodeSnippet>

Its complaining that we cant build our entire app state from a reference to an empty tuple.
Such a trait bound existing makes no sense.

It turns out its originating from a bug in the debug_handler macro.
It expects the first argument to be something like State\<T> so AppState seems to 
be messing with it. It turns out we don't need to wrap our appstate in state so we can 
just remove that.

<CodeSnippet filename="main.rs">
    ```rust 
        type AppState = Arc<Mutex<rusqlite::Connection>>;

        // in main()
            let state = Arc::new(Mutex::new(db));
        ...

        async fn kv_set(state : State<AppState>, req : Json<KVRequest>) -> impl IntoResponse
        {
            let cnxn = state.lock().await;
            db_set(&cnxn, &req.key, &req.value).map_err(|e| anyhow!(e))?;
            return Ok(());
        }
    ```
</CodeSnippet>

That should fix it but we have a new error now. 

<CodeSnippet filename="error">
    ``` 
        the method `anyhow_kind` exists for reference `&lib::SetError`, but its trait bounds were not satisfied
        the following trait bounds were not satisfied:
        `lib::SetError: Into<anyhow::Error>`
        which is required by `lib::SetError: anyhow::kind::TraitKind`
        `lib::SetError: std::fmt::Display`
    ```
</CodeSnippet>

This one is a can we kicked down the road for brevity earlier when we defined SetError.  
Any decent error type needs to be convertible to a string.
In rust, to_string() is provided by the Display trait.
Anyhow expects Display to exist on anything we're giving it as an error for that purpose but 
we never implemented Display on SetError. 

<CodeSnippet filename="lib.rs">
    ``` rust 
        use std::fmt::Display;
        ...
        impl Display for SetError 
        {
            fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
                write!(f, "{:?}", self)
            }
        }
    ```
</CodeSnippet>

This is the laziest possible implementation of Display - it actually constitutes a vulnerability in this use case, because 
we'll be sending our full error messages to the client which is not good. 

Proper practice is to send those to the console and just give the client what it needs to know.

<CodeSnippet filename="lib.rs">
    ``` rust 
        impl Display for SetError 
        {
            fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
                match self 
                {
                    Self::Oversized => write!(f, "Error: Oversized Key Or Value"),
                    Self::KeyAlreadyPresent => write!(f, "Error: Key Already Present"),
                    Self::DBError(e) => {println!("{:?}", e); write!(f, "Internal Server Error")}
                }
            }
        }
    ```
</CodeSnippet>

Thats better, but the 'proper'est way would probably be to use the tracing library.
Axum's github has plenty of examples integrating it, and its included by default in their tutorial.

<CodeSnippet filename="main.rs">
    ``` rust 
        #[derive(Deserialize)]
        struct KVSetRequest {key : String, value: String}

        #[debug_handler]
        async fn kv_set(state : State<AppState>, req : Json<KVSetRequest>) -> Result<impl IntoResponse, String>
        {
            let cnxn = state.lock().await;
            db_set(&cnxn, &req.key, &req.value).map_err(|e| e.to_string())?;
            return Ok(());
        }
    ```
</CodeSnippet>

We dont actually need the anyhow! macro anymore, as String implements IntoResponse directly.

Now that we have set done, get is trivial - mostly a copy paste. 

<CodeSnippet filename="main.rs">
    ``` rust
        #[derive(Deserialize)]
        struct KVGetRequest {key : String}

        #[debug_handler]
        async fn kv_get(state: State<AppState>, req: Json<KVGetRequest>) -> Result<impl IntoResponse, String>
        {
            let cnxn = state.lock().await;
            let entry = db_get(&cnxn, &req.key).map_err(|e| e.to_string())?;
            return Ok(Json(entry));
        }
    ```
</CodeSnippet>

And with that the server side is fully functional!
