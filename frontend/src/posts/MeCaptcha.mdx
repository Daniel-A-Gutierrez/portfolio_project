# Dealing with Bots
Web scrapers gonna scrape, but I want to make sure that the only things touching my API endpoints are humans using 
the GUI of the site, however, I don't wanna rely on CAPTCHA.  
I'll probably have a unique hidden session key specifically for testing.  
Id rather make my own, mostly for fun.  
I don't expect it to stop a dedicated adversary from accessing the API, I just want it to require human effort. If that 
effort comes in the form of designing a bot specifically to tackle the challenge that counts too.   
I also want to do it without relying on images or forms of media other than text.   

It should go like this :  
1. GUI component loads another challenge component into itself from the server randomly on page load. 
2. GUI Component issues a challenge
3. The user input is sent to the server
4. the server checks it, and issues a session token cookie if its correct, allowing some api use. 
5. the masked component is revealed and made available for use. 

For 1 I think I need a /challenges/\{slug\} (sortof) dynamic route like I use for posts. Each one will be a .astro file 
containing just the challenge. On page load I'll have the component request a random one from the server, which will send 
the rendered component through an api route. 

# First steps 
- Lets make a dumb challenge - just click a button.

- The dumb component - holds a ui element and hides it behind a button that says "Prove Your Humanity",

What I need now is a route to obtain a session token that generates a random u128 and adds it as a cookie in hexadecimal,
as well as a middleware function that checks for a valid session cookie. 

I also need a way of removing session cookies at a scheduled future time...I think I can do that with tokio::time::sleep.

Need the cookie feature of axum and also rand.

<CodeSnippet filename="main.rs::get_session"> 
    ```rust 
        use axum_extra::extract::cookie::{Cookie, CookieJar};
        use rand::prelude::*;

        #[derive(Deserialize)]
        struct SessionRequest
        {
            answer:    String,
            challenge: String,
        }

        #[debug_handler]
        async fn get_session(state: State<AppState>,
                            mut jar: CookieJar,
                            req: Json<SessionRequest>)
                            -> Result<impl IntoResponse, String>
        {
            //if theres already a session cookie and its in the map, delete it from the map.
            if let Some(old_cookie) = jar.get("session_id")
            {
                let old_cookie : u128 = u128::from_str_radix(old_cookie.value(), 16)
                    .map_err(|_| "Session ID must be hexadecimal")?;
                state.sessions.remove(&old_cookie);
            }

            if req.answer == "please" 
            {
                let new_session : u128 = rand::rng().random::<u128>();
                jar = jar.add(Cookie::new("session_id", format!("{:X}", new_session)));
                return Ok((StatusCode::OK, jar));
            }
            return Ok((StatusCode::FORBIDDEN, jar));
        }
    ```
</CodeSnippet>

Ok heres our foolproof authentication handler!  
Cookie jar is the only new part, and theres a small trick to it - jar.add doesnt mutate the jar, it 
copies it and returns a new one. Also, it needs to be returned in the response or the client won't be able 
to see any cookies you set. 

from_str_radix and [format!](https://doc.rust-lang.org/std/fmt/) are from the stdlib.

Oddly this old version of axum was really finnicky with the return types, and wouldn't accept a status code 
alone as a response. 

Now that theres a way to request a new session, I want a middleware that can check if that cookie exists
and is in the map before running any api functions.

<CodeSnippet filename="main.rs::check_session"> 
    ```rust 
        async fn check_session(state: State<AppState>,
                            jar: CookieJar,
                            req: Request<Body>,
                            next: Next<Body>)
                            -> impl IntoResponse
        {
            if let Some(cookie) = jar.get("session_id")
            {
                if let Ok(session_id) = u128::from_str_radix(cookie.value(), 16)
                {
                    if state.sessions.contains(&session_id)
                    {
                        return next.run(req).await.into_response();
                    }
                }
                else
                {
                    return (StatusCode::UNAUTHORIZED, jar, "Session ID must be hexadecimal").into_response();
                }
            }
            //its testy about just giving back the status code.
            return (StatusCode::UNAUTHORIZED, jar, "").into_response();
        }
    ```
</CodeSnippet>

This is what a [middleware](https://docs.rs/axum/latest/axum/middleware/index.html) function looks like.  
It'll run before and potentially after the handler, in a nested fashion with the bottom most middleware in the
call stack wrapping everything above it and running first. 

It expects at a minimum to take a request and potentially pass it to the next layer in the 'tower' through
next.run(req). It may also take whatever response that returns and operate on it before returning it up to the 
prior layer below it in the call stack.

<CodeSnippet filename="main.rs"> 
    ```rust 
        let api_router = axum::Router::new().route("/kv_set", put(kv_set))
                                            ...
                                            .layer(from_fn_with_state(state.clone(), check_session))
                                            .route("/get_session", post(get_session))
                                            .with_state(state);
    ```
</CodeSnippet>

This way get_session isnt protected by the authorization layer.  

The easiest way to write middleware functions is using axum's 
[from_fn](https://docs.rs/axum/latest/axum/middleware/fn.from_fn.html) and 
[from_fn_with_state](https://docs.rs/axum/latest/axum/middleware/fn.from_fn_with_state.html) helpers.

        const SERVER_URL: &'static str = "http://127.0.0.1:3334";

        fn make_client() -> Client
        {
            let client =
                reqwest::ClientBuilder::new().cookie_store(true)
                                            .build()
                                            .unwrap();
            //specify common headers, test different http modes, etc.
            return client;
        }

        enum Method
        {
            GET,
            POST,
            PUT,
            DELETE,
        }
        fn client_method(c: &Client, m: Method, url: &str) -> RequestBuilder
        {
            match m
            {
                Method::GET => c.get(url),
                Method::POST => c.post(url),
                Method::PUT => c.put(url),
                Method::DELETE => c.delete(url),
            }
        }

        async fn api_request<T>(c: &Client, route: &str, m: Method, body: &T) -> Result<Response>
            where T: ?Sized + Serialize
        {
            let req = client_method(c,m,& (format!("{}/api/{}", SERVER_URL, route)))
                        .header(CONTENT_TYPE, "application/json")
                        .body(to_json(body).unwrap());
            let res = req.send().await?; 
            return Ok(res);
        }

        #[test]
        async fn e2e() -> Result<()>
        {

                                                                                        .await?;

            return Ok(());
