# Benchmark Plan 
I plan to run 3 test clients against an axum server limited to 4 threads. 

1. A reqwest.rs client written in rust 
2. A multithreaded python client using pyreqwest 
3. A singlethreaded python client using pyreqwest 

Pyreqwest itself is a performant zero-copy wrapper around reqwest.rs, it outperforms 
urllib by a healthy ~150% for small requests and ~25% for large requests
according to their [benchmarks](https://github.com/MarkusSintonen/pyreqwest/blob/main/docs/benchmarks.md). 

More importantly it controls the variable of 'everything that isn't python', so we can really see 
the difference that python's multithreading makes. 

As for the actual test - I'm thinking I can do some sort of find-and-replace api.
Each python coroutine will read a fixed string from some file, and choose 2 substrings from that file 
of equal length. It'll then send a request to the server that looks like this:  

<CodeSnippet>
```typescript
    let request = {find: string, replace: string, text: string}; 
    //and get a response like 
    let response = string;
    //feel like theres not much sense in json'ing a single string.
```
</CodeSnippet>

Since the api uses json we'll also make use of [orjson](https://pypi.org/project/orjson/) for a claimed 500% speedup in json serialization
in python.

# Multithreading 

Now for the interesting part - lets test out python's new 'free threads'.


<CodeSnippet filename="threads">
```python 
    async def bench_threads(threads,concurrency):
        print("Benching Multithreaded Coroutines")
        text = load_text('words_100.txt')
        handles = []
        start = time.time()
        for t in range(threads):
            handles.append(asyncio.to_thread( lambda: make_thread(str(text), concurrency) ))
        await asyncio.gather(*handles)
        elapsed = time.time()-start
        print('%4.3fk req/s'%(((concurrency*threads)/elapsed)))

    #im only doing 4 threads, hence 25 samples.
    def make_thread(text,concurrency):
        async def thread():
            client = make_client()
            coroutines = []
            for i in range(concurrency):
                coroutines.append(make_coroutine(client, str(text)))
            return await asyncio.gather(*coroutines)
```

the Global Interpreter Lock disabled by default. It needs to be compiled from source

But even with that, things still failed to run. I ended up adapting the code to use ye olde python 
threading library instead of asyncio threads.

Here's the new code.

<CodeSnippet filename="multithreaded coroutines">
```python 
    import threading
    async def bench_threads(threads,concurrency):
        print("Benching Multithreaded Coroutines")
        text = load_text('words_100.txt')
        handles = []
        start = time.time()
        for t in range(threads):
            handle = threading.Thread(target=make_thread, args=(str(text),concurrency))
            handle.start()
            handles.append(handle)
        for h in handles:
            h.join()
        elapsed = time.time()-start
        print('%4.3fk req/s'%(((concurrency*threads)/elapsed)))

    #im only doing 4 threads, hence 25 samples.
    def make_thread(text,concurrency):
        async def thread():
            client = make_client()
            tasks = []
            for i in range(concurrency):
                tasks.append(make_coroutine(client, str(text)))
            await asyncio.gather(*tasks)
        asyncio.run(thread())
```
</CodeSnippet>

# Random Is The Issue?
I did some research and according to [this post](https://stackoverflow.com/questions/10021882/make-the-random-module-thread-safe-in-python)
random is the issue. It shares a single mutex between all the threads and coroutines so it bottlenecks them.  
I'm not going to bother putting the code here but long story short - giving each thread its own SystemRandom() didn't make a difference.  

# Conclusion

Python's type syntax as well as the improved tooling ecosystem are welcome changes to the language
but parallelism seems like its ... not very good at the moment. I think the approach is probably wrong.

I could understand not wanting to break existing code or make it racey but ... it did break my existing code,
that was designed for multithreading! I had to rewrite it anyway!

Just make atomic versions of the datastructures that are safe to be sent across threads, or force them to be 
cloned like js does for dedicated workers. Synchronizing everything has huge overhead.

Unless they can dramatically cut down on that I really don't expect this to ever be 
enabled by default. 

Perhaps its more useful in compute than in io heavy workloads? 

I imagine since data-frames from pandas etc. are written in C they probably don't 
have mutexes around them, and compiling python differently shouldnt change that.

So while you'd probably have to handle race conditions when sharing 
dataframes across threads, you'd also probably reap better performance gains than you see here.
